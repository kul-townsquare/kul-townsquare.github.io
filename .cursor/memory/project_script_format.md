# 剧本 JSON 格式事实

## 关键事实：用户是玩家，不是剧本作者

仓库维护者是 BotC 的**玩家**，不是剧本工具作者，**与剧本作者没有任何沟通渠道**。收到的 JSON 是已经由作者（用各自的工具）生成好的，直接拿来用。

**推论**：任何要求作者改工作流 / 改格式 / 用标准 ID 的方案都**不可行**，他们不会因为我们而改变。我们能做的只有：
- 在摄取端加护栏（校验 / 警告 / 规范化）
- 在内部实现做兼容性处理
- 不要试图引导生态；我们只是它的下游消费者

## 数据现状（基于 `剧本JSON/` 全量统计，2026-04-23）

- **495 个剧本** / 13,694 个身份条目 / 9.2 MB
- **96%（13,177 条）嵌入完整 `ability`** → 中文社区的默认格式
- **仅 0.3%（35 条）用 `{id}` 引用式** → 官方 Script Tool 格式在这个 fork 几乎不被使用
- **482 个 `_meta`** 头（约每个剧本 1 个）

## 身份 ID 野蛮生长

中文社区没有统一的 ID 规范。实际样本：
- `drunkbutton` / `djinnbutton` / `poisonerbutton`（官方名 + `button` 后缀，疑来自某脚本编辑器）
- `woheduolema111` / `mofashuijingqiu111` / `zhijiagedaziji111`（拼音 + `111`）
- `1_rose` / `_rose` / `31_rose`（剧本级前缀）
- `1_22` / `1_21`（`src/roles.json` 采用的 edition_id + 位置格式）

同一个经典身份（如"醉汉"）可能在不同剧本里 ID 完全不同。**跨剧本 ID 比对无效**。

## schema drift

除 README 声明的字段外，实际 JSON 里常见的扩展字段：
- `attribution`（作者/来源署名，高频）
- `flavor`（风味文本，高频）
- `sch_team`（未知用途，出现 4258 次，值得审计）
- `almanac`（`_meta` 里的 URL 字段，指向身份图鉴网页）
- `isCustom`（前端 `src/store/index.js` 里的 customRole 模板用过）

## `src/roles.json` 的真实定位

它是**内置身份库**，主要用途：
1. 提供官方 3 版本（TB / BMR / SnV）+ 旅行者 + 传奇身份的基础数据
2. 用户选"内置版本"时的默认数据源
3. 用户上传的剧本里若出现 `{id}` 引用式条目的**兜底查询**

由于中文剧本几乎从不用 `{id}` 引用，`src/roles.json` 实际上**只承载第 1 条用途**。它自身 ID 采用 `1_22` 风格，对外不暴露在用户可见的协议里（剧本文件不引用它）。

## 对开发的影响

- 不要试图"迁移到官方 Script Tool 兼容" —— 没有用户会因此受益
- 不要动 `src/roles.json` 的 ID 格式除非有清晰的内部价值
- 加功能时优先处理**摄取端兼容性**（容忍 schema drift、容忍嵌入能力、容忍未知字段）
- 护栏/校验脚本的默认行为应是 **report**，不是 **reject**（用户不能修作者的剧本）
